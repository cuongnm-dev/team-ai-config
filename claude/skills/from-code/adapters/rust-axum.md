# Adapter: `rust-axum`

Axum (0.6+) + common Rust web stacks (tokio, tower, sqlx / sea-orm / diesel).

## Detect

| Confidence | Signals |
|---|---|
| HIGH | `Cargo.toml` with `axum = "..."` AND a main.rs with `axum::Router` usage |
| MEDIUM | `Cargo.toml` has `axum` but no clear entry (library crate?) |
| LOW | Other Rust web frameworks (actix-web, rocket, warp) detected — use different adapter |
| NONE | No Rust web framework |

Scope note: this adapter primarily targets **Axum**. Actix-web / Rocket / Warp would be separate adapters (v2). Tree-sitter queries below are axum-specific; a small extension point allows overriding for other routers.

Version detection: parse `axum = { version = "X.Y.Z" }` or `axum = "X.Y.Z"`.

## adapter.yaml highlights

```yaml
id: rust-axum
display_name: Axum (Rust)
kind: backend
languages: [rust]

detect:
  file_globs: ["**/Cargo.toml", "**/src/main.rs", "**/src/lib.rs"]
  content_patterns:
    high:
      - { glob: "**/Cargo.toml", contains: "axum" }
      - { glob: "**/src/*.rs", contains: "axum::Router" }

queries:
  routes:
    language: rust
    # Router::new().route("/path", get(handler)).route(...)
    query: |
      (call_expression
        function: (field_expression
          field: (field_identifier) @method (#eq? @method "route"))
        arguments: (arguments
          (string_literal) @path
          (call_expression
            function: (identifier) @http-fn
            (#match? @http-fn "^(get|post|put|patch|delete|head|options|any)$")
            arguments: (arguments (identifier) @handler))))

  nested_routers:
    language: rust
    query: |
      (call_expression
        function: (field_expression
          field: (field_identifier) @method (#eq? @method "nest"))
        arguments: (arguments
          (string_literal) @prefix
          (call_expression) @router-expr))

  entities:
    language: rust
    # sqlx::FromRow, sea-orm entities, diesel Queryable
    query: |
      (struct_item
        (attribute_item) @attrs
        name: (type_identifier) @entity-name)
    post_process: extract.filter_entity_attrs

  middleware_auth:
    language: rust
    # .layer(middleware::from_fn(require_auth)) / .route_layer(axum_auth::Auth)
    query: |
      (call_expression
        function: (field_expression
          field: (field_identifier) @method (#match? @method "^(layer|route_layer)$"))
        arguments: (arguments) @args)

i18n:
  file_globs:
    - "**/locales/**/*.ftl"          # Fluent
    - "**/i18n/**/*.json"
    - "**/src/**/translations/**/*.json"

tests:
  file_globs:
    - "**/src/**/*.rs"   # #[test] functions inline
    - "**/tests/**/*.rs" # integration tests
  describe_query:
    language: rust
    query: |
      (function_item
        (attribute_item) @test-attr
        name: (identifier) @test-name)
```

## extract.py — key functions

### `resolve_routes(route_calls, nested_calls)`

Axum routing is builder-style:
```rust
let orders_router = Router::new()
    .route("/", get(list_orders).post(create_order))
    .route("/:id", get(get_order).put(update_order).delete(delete_order))
    .layer(middleware::from_fn(require_auth));

let app = Router::new()
    .nest("/api/v1/orders", orders_router)
    .nest("/api/v1/users", users_router);
```

Resolution:
1. Build tree of router expressions keyed by binding name (`orders_router`, `app`)
2. Resolve `.nest(prefix, sub_router)` to concatenate prefixes
3. `.route(path, method_router)` where method_router is `get(fn).post(fn)...` chain → one route per HTTP method
4. Handler symbol = function identifier
5. Final path = nest prefix chain + route path

Note: Rust's statement-based style means full resolution often needs to walk multiple `let` bindings. The adapter performs best-effort static resolution; dynamically-constructed routers (in functions accepting State) use light control-flow analysis.

### `resolve_entities(struct_matches)`

Rust has no "ORM-universal" entity marker. Inspect struct attributes:

1. **sqlx**:
   ```rust
   #[derive(sqlx::FromRow)]
   pub struct Order { id: Uuid, total: Decimal, ... }
   ```
   → entity; fields from struct body; PK inferred from `id: Uuid` or attribute `#[sqlx(primary_key)]`

2. **sea-orm**:
   ```rust
   #[derive(DeriveEntityModel)]
   #[sea_orm(table_name = "orders")]
   pub struct Model { ... }
   #[sea_orm(primary_key)] id: Uuid,
   ```
   → table_name from attribute; fields from Model struct; relations from `Relation` enum

3. **diesel**:
   ```rust
   #[derive(Queryable, Identifiable)]
   #[diesel(table_name = orders)]
   pub struct Order { id: Uuid, ... }
   ```
   → parse `schema.rs` generated by diesel CLI for authoritative column types

Migration source (preferred):
- `migrations/` folder (sqlx / diesel / refinery) — parse SQL
- `sea-orm-migration` Rust migration files — parse `Schema::create_table` calls

PII detection:
- Field name heuristics (same as other adapters)
- Attribute `#[serde(skip_serializing)]` on PII fields (common pattern)
- Type wrappers: `EncryptedString`, `Secret<T>` from `secrecy` crate

### `normalize_auth_scope()`

Axum auth is middleware-based:
```rust
.layer(middleware::from_fn(require_auth))
.layer(middleware::from_fn_with_state(state, require_admin))
```

Patterns:
- `require_auth` / `authenticate` named function → `authenticated`
- `require_admin` / `admin_only` → `role:admin` (inferred from name)
- `axum_login::login_required` → `authenticated`
- Custom extractors using `FromRequestParts` + claims parsing → inspect body for role/scope checks
- Route-level `.route_layer(...)` binds only to routes before it in chain
- Global `.layer(...)` on root Router applies to all

When auth function body does dynamic checks (DB lookup for RBAC), mark `policy:custom` with evidence pointer.

### `extract_di_graph()`

Rust has no traditional DI container. "Dependencies" are:
- Types in handler `State<T>` extractor → state edges `{from: handler, to: T, kind: state-extractor}`
- Trait implementations in handler params (dyn Traits or generic bounds)
- Axum's `Extension<T>` layers

Component boundaries come from `mod` structure + pub/private visibility. Edges derived from module import graph (`use` statements).

### `extract_integrations()`

- HTTP outbound: `reqwest`, `hyper`, `surf`, `ureq`
- MQ: `lapin` (RMQ), `rdkafka`, `async-nats`
- Cache: `redis` crate, `mobc-redis`
- Storage: `aws-sdk-s3`, `rusty-s3`, `minio-rs`
- DB: `sqlx` / `sea-orm` / `diesel` — connection string from env via `dotenvy` / `config`
- Tracing/Observability: `tracing` + `opentelemetry` exporters
- Scheduler: `tokio-cron-scheduler` `cron!` macro

### `extract_feature_flags()`

- `cfg` attributes: `#[cfg(feature = "beta")]` (compile-time flags — note, not runtime)
- Runtime config: `config` crate with feature_flags table; env-var pattern `ENABLE_X`
- LaunchDarkly / Unleash Rust SDK

## Validator-derived business rules

Rust web ecosystem uses `validator` crate heavily:
```rust
#[derive(Deserialize, Validate)]
pub struct CreateOrderRequest {
    #[validate(length(min = 1))]
    pub items: Vec<OrderItem>,

    #[validate(custom = "validate_promo")]
    pub promo_code: Option<String>,
}
```

Extract `#[validate(...)]` attributes → rules. Custom functions flagged `[CẦN BỔ SUNG: custom validator logic]` unless function body is simple enough to summarize.

## Tests

Rust inline `#[test]` and `#[tokio::test]` functions count as unit tests. Tests in `tests/` folder are integration. No `describe` blocks natively — the **function name** is the describe equivalent (e.g. `test_create_order_with_voucher_applies_discount`). Preserve verbatim.

## Limitations (v1)

- Macros that generate code (e.g. `tonic::include_proto!`) not expanded — gRPC service structure invisible
- `actix-web` / `rocket` / `warp` need dedicated adapters (not shipped v1)
- Build-time codegen (protobuf, schema files) limits entity discovery if only final artifacts are committed — rely on migrations where possible
- Complex type-level programming (associated types, GATs) may defeat entity extraction; user can supplement in interview
